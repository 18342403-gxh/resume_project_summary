# 简历项目总结 Skill

> 输入：Git 仓库路径（1 个或多个）→ 输出：三段式简历描述 + 可选贡献清单

---

## 模式判断

| 用户意图 | 模式 |
|----------|------|
| 当前项目 / 单个路径 | 单项目 |
| 多个路径 | 多项目独立 |
| 多个路径 + "合并" | 多项目合并（先跑关联性检测） |
| "亮点 / 有什么可以写" | 亮点挖掘（始终追加在输出末尾） |

---

## 执行流程

### Step 1：数据采集

对每个项目路径执行（脚本：`scripts/collect-git-data.sh`）：

```bash
AUTHOR=$(git config user.name)
# 全部提交
git log --author="$AUTHOR" --pretty=format:"%h %ad %s" --date=short
# 模块分布
git log --author="$AUTHOR" --pretty=format:"%s" | grep -oE "^[a-z]+\([^)]*\)" | sort | uniq -c | sort -rn
# 高频文件 Top40
git log --author="$AUTHOR" --pretty=format:"" --name-only | grep -v "^$" | sort | uniq -c | sort -rn | head -40
# 时间范围
git log --author="$AUTHOR" --format="%ad" --date=short | sort | sed -n '1p;$p'
```

同时读取：`package.json`、`README.md`、`src/pages/` 或 `src/views/` 目录、`git remote -v`。

---

### Step 2：多项目合并（≥2 个项目时）

脚本：`scripts/merge-projects.sh`，按 5 维度评分（详见 `config/merge-rules.yaml`）：

| 维度 | 权重 |
|------|------|
| 技术栈重叠 | 30 |
| 业务域关联 | 25 |
| 代码引用 | 20 |
| 时间线连续 | 15 |
| 组织归属 | 10 |

≥60 自动合并 / 40-59 询问用户 / <40 独立输出。

---

### Step 3：归类提炼

- 按 commit 前缀聚类为 3-6 个模块
- 排序：提交数多 → 少
- 零散 commit（样式/typo/CR）合并为"体验优化/多轮迭代"

---

### Step 4：结构化输出（严格按下方模板填写）

👇 **直接按模板输出，用实际内容替换 `{{占位符}}`，不要输出占位符本身。**

---

## 输出模板

### 模板 A：三段式（必须输出）

```
## 项目简介

{{80-180字，一段话。必须包含：行业是什么 + 做什么产品 + 用什么技术形态 + 业务价值是什么。
包装词参考 config/packaging-dictionary.yaml，选 2-3 个即可，不堆砌。
合并项目可放宽至 250 字。禁止出现无数据支撑的量化指标。}}

## 项目技术

- **{{分组名1}}**：{{技术名}} + {{解决了什么问题}}
- **{{分组名2}}**：{{技术名}} + {{解决了什么问题}}
- **{{分组名3}}**：{{技术名}} + {{解决了什么问题}}
- **{{分组名4}}**：{{技术名}} + {{解决了什么问题}}

{{规则：
- 只写 package.json 中存在的依赖，不编造
- 分组参考：跨端/状态管理/UI/桥接/工程化/监控/安全
- 每项必须是"技术 + 解决了什么"句式，不能只写技术名}}

## 个人贡献

1. **{{主力模块名}}**：{{动词（主导/负责/设计）+ 做了什么 + 覆盖什么场景/页面，配合谁完成了什么}}。
2. **{{模块2}}**：{{动词 + 做了什么 + 业务价值}}。
3. **{{模块3}}**：{{动词 + 做了什么 + 复用/覆盖范围}}。
4. **{{基础建设/协作}}**：{{配合团队完成了什么 + 规范/组件/工具沉淀}}。

{{规则：
- 3-5 条（合并项目 3-6 条）
- 每条 = 加粗模块名 + 1-2 句话
- 第一条必须是提交数最多的模块
- 至少 1 条提到团队协作
- 禁止暴露：具体文件名、CSS 属性、API 路径、单个 bugfix
- 禁止编造：无 commit 支撑的内容不写}}
```

### 模板 B：可选贡献（始终追加在模板 A 之后）

```
---

## 可选贡献（项目中有价值的事项，可酌情写入简历）

> 以下条目从项目整体架构与代码实现中提取，写法与个人贡献一致，可直接复制到简历。
> 面试时准备好"团队怎么做的、你参与了哪部分"的回答即可。

1. **{{组件/模块名}}**：{{封装/设计/打通了什么，支持什么能力，被 N 个场景复用}}。 `#组件`
2. **{{业务流程名}}**：{{端到端实现了什么流程，包含哪些状态/步骤}}。 `#业务流程`
3. **{{工具/Hook 名}}**：{{抽象了什么能力为通用方法，被 N 处引用}}。 `#工具封装`
4. **{{架构方案名}}**：{{设计了什么架构方案，解决了什么问题}}。 `#架构`
5. **{{工程化实践}}**：{{落地了什么工程化方案，支撑了什么}}。 `#工程化`
6. ⬆️ **{{已有贡献的升级版，可替换第 X 条}}**：{{更有分量的写法}}。 `#架构`

{{规则：
- 4-8 条，宏观架构层和代码实现层各占一半
- 写法：第一人称 + 具体模块名 + 业务语境，不写分析报告
- 去重：与个人贡献重叠的不重复出现（升级版用 ⬆️ 标记）
- 来源：必须基于实际存在的代码/依赖/配置
- 代码级亮点识别依据：
  · 被 ≥5 处 import 的组件/hook → "封装 XX，覆盖 N 个场景"
  · ≥200 行的业务页面 → "端到端实现 XX 流程"
  · 含状态机/多步骤的页面 → "打通 XX 全链路"
  · 有降级/重试的模块 → "设计 XX 容错方案"
  · 统一请求层 → "设计请求中间件体系"}}

---
💡 使用说明：
- 直接复制到"个人贡献"中替换或追加
- ⬆️ 条目是升级版，替换后更有分量
- 标签仅供筛选，写入简历时删除
```

---

### Step 5：自校验

生成后按 `config/validation-rules.yaml` 逐条校验，最多修正 3 轮：
- **error 级**：必须通过（字数/格式/编造/细节泄露）
- **warning 级**：尽力修复（排序/团队提及/分组）
- 校验通过 → 末尾附 `✅ 校验通过`
- 有未修复 warning → 末尾标注具体项
- 有未修复 error → 末尾标注 `❌` 并给修改建议

---

## 硬性规则（不可违反）

1. **不编造**：所有内容必须有 commit / package.json / 代码文件支撑
2. **不泄露细节**：禁止出现具体文件名、CSS 属性、API 路径、单个 bugfix
3. **中文输出**（技术名词除外）
4. **无寒暄**：直接输出三段式，不加"好的""以下是"等开场
5. **格式严格**：`## 项目简介` / `## 项目技术` / `## 个人贡献` / `## 可选贡献` 四个二级标题必须存在

---

## 交互迭代

| 用户反馈 | 操作 |
|----------|------|
| "太具体了" | 合并细节，用"体验优化/多轮迭代"代替 |
| "黑话不够" | 加入更多 config/packaging-dictionary.yaml 中的词 |
| "太多了" | 压缩为 3 条贡献 |
| "合并一下" | 将相近模块合并为一条 |
| "再加点" | 从可选贡献中选取补充 |
