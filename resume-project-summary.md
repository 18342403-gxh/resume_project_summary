# 简历项目总结 Skill

## 适用场景

- 基于一个或多个工作区，帮用户总结个人贡献，用于简历、述职、周报、晋升材料。
- 支持单项目独立输出，也支持多项目智能合并为一段经历。
- 输入：工作区源码 + Git 记录（可指定多个路径）；输出：**项目简介 + 技术栈 + 个人贡献**三段式。

---

## 模式判断

根据用户输入判断执行模式：

| 用户意图 | 模式 | 说明 |
|----------|------|------|
| 只提到当前项目 | 单项目模式 | 仅采集当前工作区 |
| 指定多个路径 | 多项目独立模式 | 逐一采集，分别输出 |
| 指定多个路径 + "合并/整合" | 多项目合并模式 | 采集后检测关联性，合并输出 |
| 提到多个路径但未明确 | 先独立采集，再建议合并 | 展示关联性分析，询问用户 |

---

## 执行流程

### Phase 1：数据采集（每个项目独立执行）

对每个目标项目路径执行以下命令。若用户未指定路径，默认使用当前工作区。

可直接调用辅助脚本：
```bash
bash scripts/collect-git-data.sh <项目路径1> [项目路径2] [项目路径3] ...
```

或手动执行：

#### 1.1 识别当前用户

```bash
git config user.name
```

#### 1.2 拉取用户的所有提交（按时间倒序）

```bash
git log --author="$(git config user.name)" --pretty=format:"%h %ad %s" --date=short
```

#### 1.3 按模块统计贡献量（基于 commit message 前缀）

```bash
git log --author="$(git config user.name)" --pretty=format:"%s" | grep -oE "^[a-z]+\([^)]*\)" | sort | uniq -c | sort -rn
```

#### 1.4 按月统计工作量

```bash
git log --author="$(git config user.name)" --pretty=format:"%ad" --date=format:"%Y-%m" | sort | uniq -c
```

#### 1.5 高频修改文件 Top 40（定位主力模块）

```bash
git log --author="$(git config user.name)" --pretty=format:"" --name-only | grep -v "^$" | sort | uniq -c | sort -rn | head -40
```

#### 1.6 提取项目元信息

- `package.json`（依赖与技术栈）
- `README.md`（项目定位）
- `src/pages/` 或 `src/views/` 目录结构（业务模块划分）
- `git remote -v`（仓库归属）
- 项目时间范围：`git log --author="..." --format="%ad" --date=short | sort | head -1` 和 `tail -1`

---

### Phase 2：关联性检测（多项目时执行）

当输入 ≥ 2 个项目时，执行关联性分析。可调用辅助脚本：
```bash
bash scripts/merge-projects.sh <项目路径1> <项目路径2> ...
```

#### 2.1 检测维度

按以下维度逐一评分，满分 100：

| 维度 | 权重 | 判断方式 |
|------|------|----------|
| 技术栈重叠 | 30 | `package.json` dependencies 交集 / 并集 |
| 业务域关联 | 25 | commit message 中提取业务关键词，计算 Jaccard 相似度 |
| 代码引用关系 | 20 | 是否存在 workspace / submodule / 相同 npm scope |
| 时间线连续性 | 15 | 项目活跃时间段是否重叠或首尾相接（间隔 ≤ 2 月） |
| 组织归属 | 10 | git remote 是否指向同一 org/group |

#### 2.2 合并决策

- **总分 ≥ 60**：强烈建议合并，直接合并输出
- **总分 40–59**：建议合并，展示理由后询问用户
- **总分 < 40**：不建议合并，独立输出

#### 2.3 合并策略

合并时选择"主项目"（提交数最多 or 时间跨度最长的项目），将其他项目的贡献归入主项目的模块分类中：

- 若子项目有独特模块（主项目无对应），新增模块条目
- 若子项目模块与主项目重叠，合并描述，注明覆盖范围扩大
- 项目简介合并为一段，体现更大的业务全景

---

### Phase 3：归类与提炼

- 按 commit message 前缀（`feat(模块)` / `fix(模块)` 等）对 commit 聚类。
- 识别出 **3–6 个主力模块**（按提交数或文件改动数排序）。
- 每个模块用"**动词 + 模块名 + 关键交付物**"概括。
- 合并零散 commit（样式调整、CR 修改、typo fix 等不单独列）。
- 识别参与度："独立主导 / 端到端负责 / 深度参与 / 配合完成"。

---

### Phase 4：结构化输出

#### ① 项目简介（互联网黑话版）

- 80–180 字，一段话（合并项目可放宽至 250 字）。
- 必须体现：**行业定位、业务价值、技术形态、战略意义**。
- 包装词参考 `config/packaging-dictionary.yaml`。

#### ② 项目技术

- 分组呈现，每项用"**技术选型 + 解决了什么**"的句式。
- 只保留 `package.json` 中真实存在的依赖，不编造。
- 合并项目时取所有项目技术栈的并集，去重后分组。

#### ③ 个人贡献

- **3–6 条**（合并项目可到 6 条），用 `1. 2. 3.` 编号。
- 每条 = **加粗模块名** + 1–2 句话（个人主导 + 配合团队）。
- 合并项目的贡献需标注覆盖的子项目范围（如"横跨 XX 和 YY 两个系统"）。
- 按"主力模块 → 次要模块 → 基础建设与协作"排序。

---

### Phase 5：自校验与修正

生成内容后，**必须**执行校验流程。校验规则详见 `config/validation-rules.yaml`。

#### 5.1 校验执行顺序

```
全局校验(global) → 项目简介(intro) → 项目技术(tech) → 个人贡献(contribution) → 合并专项(merge_specific)
```

#### 5.2 校验规则摘要

**全局校验：**
| ID | 规则 | 等级 |
|----|------|------|
| GLOBAL_CHINESE | 输出必须为中文（技术名词除外） | error |
| GLOBAL_NO_GREETING | 不得包含寒暄/开场白/结束语 | error |
| GLOBAL_MARKDOWN_FORMAT | 三个二级标题必须存在且层级正确 | error |
| GLOBAL_CONSISTENCY | 简介中的业务模块应在贡献中有对应 | warning |

**项目简介校验：**
| ID | 规则 | 等级 |
|----|------|------|
| INTRO_LENGTH | 单项目 80-180 字，合并项目 80-250 字 | error |
| INTRO_NO_METRICS | 不得包含无数据支撑的量化指标 | error |
| INTRO_FOUR_ELEMENTS | 必须体现行业定位/业务价值/技术形态/战略意义（至少 3 项） | error |
| INTRO_NO_STACK_DUMP | 不应在简介中堆砌 ≥3 个技术框架名 | warning |

**项目技术校验：**
| ID | 规则 | 等级 |
|----|------|------|
| TECH_REAL_DEPS | 所有技术选型必须在 package.json 中有据可查 | error |
| TECH_FORMAT | 每项必须是"技术选型 + 解决了什么"句式 | error |
| TECH_NO_ORPHAN | 不应出现与个人贡献完全无关的技术项 | warning |
| TECH_GROUPING | 技术项应按维度分组 | warning |

**个人贡献校验：**
| ID | 规则 | 等级 |
|----|------|------|
| CONTRIB_COUNT | 单项目 3-5 条，合并项目 3-6 条 | error |
| CONTRIB_NO_DETAIL_LEAK | 不得暴露过于具体的实现细节（文件名/CSS 属性/API 路径） | error |
| CONTRIB_NO_FABRICATION | 贡献内容必须有 commit 或文件记录支撑 | error |
| CONTRIB_FORMAT | 每条必须是"加粗模块名 + 1-2 句描述"格式 | error |
| CONTRIB_TEAM_MENTION | 至少 1 条提到团队协作 | warning |
| CONTRIB_PRIORITY_ORDER | 应按贡献量从大到小排序 | warning |
| CONTRIB_NO_OVERLAP | 各条目间不应有明显内容重复 | warning |

**合并项目专项校验（仅合并模式）：**
| ID | 规则 | 等级 |
|----|------|------|
| MERGE_SOURCE_MARK | 贡献应标注覆盖的系统范围 | warning |
| MERGE_NO_DUPLICATE | 不应出现两条描述同一件事的贡献 | error |
| MERGE_PRIMARY_CLEAR | 简介应是统一视角，非简单拼接 | warning |

#### 5.3 校验执行逻辑

```
初始生成 → 第 1 轮校验
  ├─ 存在 error → 修正 → 第 2 轮校验
  │     ├─ 仍有 error → 修正 → 第 3 轮校验（最终轮）
  │     │     ├─ 仍有 error → 输出并标注"⚠️ 以下规则未通过，请人工确认"
  │     │     └─ 全部通过 → 输出
  │     └─ 全部通过 → 输出
  └─ 无 error → 检查 warning
        ├─ 存在 warning → 尝试修正 1 轮 → 输出（仍存在的 warning 标注给用户）
        └─ 无 warning → 直接输出
```

#### 5.4 校验结果展示

校验通过时，在输出末尾附加：

```
---
✅ 校验通过（共检查 X 条规则，全部合格）
```

校验有 warning 未修复时：

```
---
⚠️ 校验完成（X 条通过，Y 条 warning 未修复）
- [RULE_ID]: [具体说明]
```

校验有 error 未修复时（极端情况）：

```
---
❌ 校验未完全通过（以下规则需人工确认）
- [RULE_ID]: [具体说明 + 建议修改方向]
```

## 输出模板

### 单项目模板

```markdown
## 项目简介

[互联网黑话包装的 80–180 字介绍]

## 项目技术

- **跨端架构**：[技术选型] + [解决了什么问题]
- **状态管理**：...
- **UI 体系**：...
- **工程化**：...

## 个人贡献

1. **[主力模块]**：[独立主导的核心交付 + 业务价值]。
2. **[模块 2]**：[端到端负责的能力 + 覆盖场景]。
3. **[模块 3]**：[参与 + 配合的工作 + 一致性收益]。
4. **[基础/横向工作]**：[跨端适配、桥接改造、工程规范等]。
```

### 合并项目模板

```markdown
## 项目简介

[合并后的全景描述，150–250 字，体现多系统协同的业务价值]

## 项目技术

- **跨端架构**：...
- **微前端/模块联邦**：...
- **状态管理**：...
- **UI 体系**：...
- **工程化与 DevOps**：...

## 个人贡献

1. **[核心模块 - 横跨系统 A/B]**：[统一的能力描述 + 多系统覆盖]。
2. **[主力模块]**：[独立主导 + 业务成果]。
3. **[模块 3]**：[端到端负责 + 场景]。
4. **[基础设施]**：[跨项目复用的基建工作]。
5. **[协作与规范]**：[推动多项目统一的规范/流程]。
```

## 交互规则

1. **模式确认**：多项目场景下，先展示关联性分析结果，再确认"合并 or 独立"。
2. **迭代式输出**：首次输出偏完整；用户说"太多/太具体/合并"时压缩；用户说"黑话不够/再包装"时加强。
3. **双向校准**：若用户提问模糊，先默认产出三段式，再询问是否需要针对特定模块深挖。
4. **不虚构**：所有模块、技术选型必须有 commit 或 `package.json` 依据。无法证实的成绩（如"性能提升 30%"）**禁止编造**。
5. **中文输出**，代码块只用于命令展示。
6. **路径灵活**：用户可传入绝对路径、相对路径、或 `~/projects/*` 通配符。

## 反例（不要这样写）

- ❌ "修复了车辆绑定页字体大小"——太具体，应合并为"完成车辆绑定模块体验优化"。
- ❌ "DAU 提升 50%"——无数据支撑，禁止编造。
- ❌ 把 100+ commit 全部列出——应聚类到 3–6 个模块。
- ❌ 只写自己做的，完全不提团队——适度带"配合团队完成 XX"体现协作。
- ❌ 两个强关联项目分开写导致内容重复——应合并为一段更有分量的经历。
- ❌ 强行合并无关项目——技术栈和业务完全不同的项目不应合并。
